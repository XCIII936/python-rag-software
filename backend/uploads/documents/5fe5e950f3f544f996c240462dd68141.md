# 第七章 软件质量与过程改进

## 学习目标
- 理解软件质量的定义与质量模型
- 理解软件质量保证（SQA）的活动内容
- 掌握 CMMI 能力成熟度模型的基本概念
- 了解 ISO 9000/25010 等质量标准
- 理解软件度量与过程改进的基本方法

---

## 7.1 软件质量概述

### 7.1.1 软件质量的定义
软件质量是软件与明确说明的功能和性能需求、明确文档化的开发标准以及所有专业开发的软件所期望的隐含特征相一致的程度。

简言之，软件质量包含三个层次：
1. 软件需求是衡量软件质量的基础
2. 开发标准定义了一组用于指导软件开发方式的准则
3. 用户对软件的隐含需求（即使未明确提出，用户也期望满足，如易用性）

### 7.1.2 软件质量的特性模型（ISO/IEC 25010）
ISO/IEC 25010 定义了软件产品质量的 8 大特性：
1. **功能适合性（Functional Suitability）**
2. **性能效率（Performance Efficiency）**
3. **兼容性（Compatibility）**
4. **易用性（Usability）**
5. **可靠性（Reliability）**
6. **安全性（Security）**
7. **可维护性（Maintainability）**
8. **可移植性（Portability）**

## 7.2 软件质量保证（SQA）

### 7.2.1 SQA 的定义
软件质量保证是为了保证软件产品和过程完全符合既定的标准和过程而建立的一系列有计划、有组织的活动。

### 7.2.2 SQA 的主要活动
1. 制定质量保证计划
2. 参与需求评审、设计评审、代码评审
3. 监督软件过程是否遵循规定的标准和流程
4. 缺陷跟踪与分析
5. 度量数据的收集与分析

### 7.2.3 质量保证 vs 质量控制
- **质量保证（QA, Quality Assurance）**：面向过程，通过规范化流程预防缺陷产生（如评审、培训）
- **质量控制（QC, Quality Control）**：面向产品，通过测试等手段发现并纠正缺陷

## 7.3 CMMI 能力成熟度模型

### 7.3.1 CMMI 概述
CMMI（Capability Maturity Model Integration）是评估和改进组织软件过程能力的模型框架，由美国卡内基梅隆大学软件工程研究所（SEI）开发。

### 7.3.2 CMMI 的五个成熟度等级
| 等级 | 名称 | 特征 |
|------|------|------|
| 1级 | 初始级（Initial） | 过程无序，成功依赖个人英雄主义 |
| 2级 | 已管理级（Managed） | 建立了基本的项目管理流程（需求管理、项目计划、配置管理等） |
| 3级 | 已定义级（Defined） | 过程已经标准化、文档化，并在组织范围内推广 |
| 4级 | 定量管理级（Quantitatively Managed） | 采用定量方法度量和控制过程 |
| 5级 | 优化级（Optimizing） | 持续改进过程，基于度量数据主动优化 |

### 7.3.3 CMMI 的过程域示例（以2级为例）
- 需求管理（REQM）
- 项目计划（PP）
- 项目监控（PMC）
- 配置管理（CM）
- 过程和产品质量保证（PPQA）

## 7.4 ISO 质量标准

### 7.4.1 ISO 9000 系列
ISO 9000 是国际通用的质量管理体系标准，强调"以顾客为中心"、"过程方法"、"持续改进"等质量管理原则。ISO 9001 是可供第三方认证的具体要求标准。

### 7.4.2 ISO/IEC 25010（SQuaRE 系列）
如 7.1.2 节所述，规定了软件产品质量的评价模型，是当前软件质量评价的主流国际标准，取代了较早的 ISO/IEC 9126。

### 7.4.3 ISO/IEC 12207
定义了软件生命周期过程的国际标准，涵盖从需求、设计、实现到维护、废弃的全过程。

## 7.5 软件度量

### 7.5.1 度量的分类
1. **产品度量**：代码行数（LOC）、圈复杂度、代码重复率、缺陷密度
2. **过程度量**：缺陷发现率、平均修复时间、迭代速度（Velocity）
3. **项目度量**：进度偏差、成本偏差、资源利用率

### 7.5.2 常见质量度量指标
- **缺陷密度（Defect Density）** = 缺陷数量 / 代码规模（KLOC）
- **缺陷检出率** = 某阶段发现缺陷数 / 总缺陷数
- **平均故障间隔时间（MTBF）**：衡量系统可靠性
- **平均修复时间（MTTR）**：衡量系统可维护性/可恢复性

### 7.5.3 GQM 方法（Goal-Question-Metric）
一种自顶向下建立度量体系的方法：
```
目标（Goal）→ 提出问题（Question）→ 确定度量指标（Metric）
```
例如：目标"提高代码质量" → 问题"缺陷是否在减少？" → 度量"每千行代码缺陷数"

## 7.6 过程改进

### 7.6.1 PDCA 循环
戴明环（PDCA）是持续改进的经典方法论：
```
Plan（计划）→ Do（执行）→ Check（检查）→ Act（处理/改进）→ 循环
```

### 7.6.2 六西格玛（Six Sigma）
一种以数据驱动、追求消除缺陷的质量管理方法，核心流程为 **DMAIC**：
```
Define（定义）→ Measure（度量）→ Analyze（分析）→ Improve（改进）→ Control（控制）
```

### 7.6.3 敏捷回顾会议
敏捷开发中通过每个迭代结束的"回顾会议（Retrospective）"，持续反思团队协作和过程中的问题，是轻量级的过程改进实践。

---

## 本章小结
1. 软件质量是多维度的概念，ISO/IEC 25010 给出了系统化的质量特性模型
2. CMMI 提供了组织级软件过程能力的成熟度评估框架，从"无序"逐步走向"持续优化"
3. 软件度量是过程改进的数据基础，GQM 方法帮助我们建立目标导向的度量体系
4. PDCA 循环是贯穿质量管理和过程改进活动的通用方法论
