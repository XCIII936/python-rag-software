# 第一章 软件工程概述

## 学习目标
- 理解软件、软件工程的定义与软件的特性
- 理解"软件危机"产生的原因及其表现
- 掌握主流软件过程模型的原理、优缺点及适用场景
- 了解软件工程的知识体系与发展趋势

---

## 1.1 软件与软件工程

### 1.1.1 什么是软件
软件（Software）是计算机系统中与硬件相互依存的部分，包括：
- **程序（Program）**：按照事先设计的功能和性能要求执行的指令序列
- **数据（Data）**：使程序能正常操纵信息的数据结构
- **文档（Documentation）**：与程序开发、维护和使用有关的图文材料

软件的主要特性：
1. **逻辑实体，而非物理实体**——具有抽象性
2. **生产方式不同于硬件**——软件是"开发"或"设计"出来的，而非传统意义上的"制造"
3. **维护方式不同于硬件**——软件维护并非老化，而是不断适应需求变更（涉及知识老化、环境变化）
4. **复杂性高**——规模庞大，逻辑关系复杂
5. **成本高、可复制性强**——研发成本高，但复制成本几乎为零

### 1.1.2 软件工程的定义
IEEE 定义：软件工程是将系统化的、规范化的、可度量的方法应用于软件的开发、运行和维护，即将工程化方法应用于软件中。

软件工程的三要素：**方法（Methods）+ 工具（Tools）+ 过程（Procedure）**

## 1.2 软件危机

### 1.2.1 软件危机的表现
20 世纪 60 年代末，随着软件规模和复杂度的增加，暴露出以下问题：
- 项目开发成本和进度估计不准确，经常超支、延期
- 用户对已完成的软件系统不满意的现象经常发生
- 软件产品质量低劣，可靠性差
- 软件通常没有适当的文档资料
- 软件成本在计算机系统总成本中所占的比例逐年上升
- 软件开发生产率提高的速度远远跟不上计算机应用普及的速度

### 1.2.2 软件危机产生的原因
1. **用户需求不明确**——需求分析不充分或需求频繁变更
2. **缺乏正确的理论和方法指导**——早期软件开发依赖个人技巧，缺乏工程规范
3. **软件规模越来越大**——系统复杂度呈指数级增长
4. **软件复杂度越来越高**
5. **缺乏有效的项目管理手段**

### 1.2.3 解决途径
针对软件危机，人们提出了工程化的思想来指导软件开发，这就是"软件工程"学科诞生的直接原因。1968 年北大西洋公约组织（NATO）在联邦德国召开会议，首次提出"软件工程"概念。

## 1.3 软件过程模型

软件过程（Software Process）是为获得高质量软件所需要完成的一系列任务的框架，规定了完成各项任务的工作步骤。

### 1.3.1 瀑布模型（Waterfall Model）
**核心思想**：将软件生命周期划分为若干阶段，依次执行、线性推进：
```
需求分析 → 设计 → 编码 → 测试 → 运行维护
```

**特点**：
- 阶段间具有顺序性和依赖性
- 强调文档驱动，每个阶段都要产出评审通过的文档
- 前一阶段的输出是后一阶段的输入

**优点**：
- 结构清晰，便于管理和控制
- 强制阶段评审，有利于保证质量
- 适合需求明确、变化较少的项目

**缺点**：
- 缺乏灵活性，难以适应需求变化
- 用户只有到项目末期才能看到成果，风险发现较晚
- 各阶段严重依赖文档，工作量大

### 1.3.2 增量模型（Incremental Model）
将软件产品分解为一系列的增量构件来设计、编码、集成和测试，每个增量构件都由多个模块组成的代码和数据构成。

**优点**：能够在较短时间内向用户提交可完成部分工作的产品，逐步完善。

### 1.3.3 螺旋模型（Spiral Model）
结合瀑布模型和原型模型（迭代）的优点，并增加了**风险分析**。每一个螺旋周期包含四个阶段：
1. 制定计划
2. 风险分析
3. 实施工程
4. 用户评估

**适用场景**：庞大、复杂并具有高风险的系统。

### 1.3.4 敏捷开发（Agile Development）
敏捷方法强调：
- **个体和互动** 胜过 过程和工具
- **工作的软件** 胜过 详尽的文档
- **客户合作** 胜过 合同谈判
- **响应变化** 胜过 遵循计划

常见敏捷方法：
- **Scrum**：以短周期迭代（Sprint，通常 1-4 周）交付可用产品增量，包含产品负责人（PO）、Scrum Master、开发团队三种角色，每日站会、迭代评审、迭代回顾等仪式
- **极限编程（XP）**：强调结对编程、测试驱动开发（TDD）、持续集成、简单设计
- **看板（Kanban）**：可视化工作流，限制在制品数量（WIP），持续交付

### 1.3.5 统一软件开发过程（RUP）
以用例驱动、以架构为中心、迭代及增量的开发过程，分为初始、细化、构建、交付四个阶段。

---

## 1.4 软件过程模型对比

| 模型 | 核心特点 | 适用场景 |
|------|---------|---------|
| 瀑布模型 | 线性顺序、文档驱动 | 需求明确、稳定的项目 |
| 增量模型 | 分批交付 | 需求可分解、需要尽早看到部分成果 |
| 螺旋模型 | 风险驱动、迭代 | 大型、高风险项目 |
| 敏捷开发 | 快速迭代、拥抱变化 | 需求不确定、需要快速响应市场 |
| RUP | 用例驱动、架构为中心 | 中大型、面向对象项目 |

---

## 本章小结
1. 软件工程诞生于解决"软件危机"的迫切需求
2. 软件过程模型指导着软件项目的组织方式，不同模型适用于不同的项目特征
3. 现代软件开发日益倾向于敏捷、迭代式的开发方式，以应对快速变化的需求

1. 为什么说软件不同于硬件，其"维护"具有特殊含义？

3. 结合实际项目案例，说明螺旋模型中风险分析环节的重要性。
