# 第六章 软件维护与项目管理

## 学习目标
- 理解软件维护的类型与特点
- 理解软件配置管理的基本概念
- 掌握项目计划与估算的常用方法
- 理解软件项目风险管理的流程
- 了解团队协作与项目管理工具

---

## 6.1 软件维护

### 6.1.1 软件维护的定义
软件维护是指在软件交付使用后，为了改正错误、适应环境变化或改进性能而对软件产品进行修改的过程。软件维护通常占软件生命周期总成本的 **60%~80%**。

### 6.1.2 软件维护的类型
1. **改正性维护（Corrective Maintenance）**：修改在测试阶段未被发现的软件错误，占比约 21%
2. **适应性维护（Adaptive Maintenance）**：为适应外部环境变化（如操作系统升级、硬件更换、法规变化）而进行的修改，占比约 25%
3. **完善性维护（Perfective Maintenance）**：为扩充功能、改善性能而进行的修改，占比最大，约 50%
4. **预防性维护（Preventive Maintenance）**：为提高软件的可维护性、可靠性而进行的预防性修改，占比约 4%

### 6.1.3 影响软件可维护性的因素
- **可理解性**：代码结构清晰、命名规范、文档齐全
- **可测试性**：便于验证修改是否正确
- **可修改性**：低耦合的设计有利于修改
- **可移植性**：便于在不同环境中运行

### 6.1.4 软件再工程
对既存系统进行重新开发，一般包括：逆向工程（从代码还原设计/需求）、重构、正向工程（依据新的设计重新实现）。

## 6.2 软件配置管理（SCM）

### 6.2.1 什么是软件配置项（SCI）
软件配置项包括所有在软件工程过程中产生的信息，如需求文档、设计文档、源代码、测试用例、用户手册等。

### 6.2.2 配置管理的主要活动
1. **配置标识**：确定需要受控的配置项，并给予唯一标识
2. **版本控制**：管理配置项的演化历史（借助 Git 等工具）
3. **变更控制**：通过变更控制委员会（CCB）评审变更请求
4. **配置状态报告**：记录和报告配置项的状态
5. **配置审计**：验证配置项是否符合规定要求

### 6.2.3 基线（Baseline）
基线是软件生命周期中各开发阶段末尾的一个特定点，一旦建立基线，后续的修改需要经过正式变更流程，如"需求基线"、"设计基线"。

## 6.3 软件项目计划与估算

### 6.3.1 项目计划的内容
- 项目范围界定
- 工作分解结构（WBS, Work Breakdown Structure）
- 进度安排（甘特图、里程碑）
- 资源分配
- 预算

### 6.3.2 常用估算方法
1. **专家判断法（Delphi 法）**：多位专家匿名独立估算，反复迭代收敛
2. **类比估算法**：参考历史类似项目的数据进行估算
3. **功能点分析法（FPA, Function Point Analysis）**：根据系统的功能点数量估算工作量，与编程语言无关
4. **COCOMO 模型（Constructive Cost Model）**：基于代码行数（KLOC）估算工作量与工期，公式如：
   ```
   工作量 E = a × (KLOC)^b   （基本 COCOMO 模型）
   ```
5. **三点估算法（PERT）**：
   ```
   期望工期 = (乐观估计 + 4×最可能估计 + 悲观估计) / 6
   ```

### 6.3.3 进度管理工具
- **甘特图（Gantt Chart）**：以横道图展示任务的起止时间及依赖关系
- **关键路径法（CPM, Critical Path Method）**：找出决定项目最短完成时间的任务序列（关键路径），关键路径上任务的延误会直接导致项目延期
- **PERT 网络图**：用于处理任务工期不确定的情况

## 6.4 软件项目风险管理

### 6.4.1 风险的分类
1. **项目风险**：影响进度或资源的风险（如人员流失、需求变更）
2. **技术风险**：影响质量或按期交付的风险（如技术方案不成熟）
3. **商业风险**：影响项目/产品生存力的风险（如市场变化、预算削减）

### 6.4.2 风险管理流程
```
风险识别 → 风险分析（评估概率和影响） → 风险规划（应对策略） → 风险监控
```

### 6.4.3 常见风险应对策略
- **规避（Avoid）**：改变计划以消除风险
- **减轻（Mitigate）**：采取措施降低风险发生概率或影响程度
- **转移（Transfer）**：将风险转移给第三方（如购买保险、外包）
- **接受（Accept）**：不采取行动，制定应急预案

### 6.4.4 风险暴露度（Risk Exposure）
```
风险暴露度 = 风险发生概率 × 风险损失（影响程度）
```
用于对风险进行优先级排序，优先处理高暴露度风险。

## 6.5 团队协作与项目管理

### 6.5.1 常见团队角色
- **项目经理（PM）**：负责项目整体计划、协调、风险控制
- **产品经理（Product Owner）**：负责需求优先级排序，代表用户利益
- **Scrum Master**：敏捷团队中的引导者，负责移除障碍、维护敏捷流程
- **架构师**：负责技术方案设计
- **开发/测试工程师**：负责具体实现和质量保障

### 6.5.2 敏捷项目管理实践
- **每日站会（Daily Standup）**：15分钟内同步进度、暴露阻塞
- **迭代计划会（Sprint Planning）**：确定本迭代要完成的任务
- **迭代评审（Sprint Review）**：演示本迭代成果，收集反馈
- **迭代回顾（Sprint Retrospective）**：总结经验教训，持续改进

### 6.5.3 常用项目管理工具
- 任务/看板管理：Jira、Trello、禅道
- 文档协作：Confluence、语雀、飞书文档
- 沟通协作：Slack、钉钉、企业微信

## 本章小结
1. 软件维护占据了软件生命周期的大部分成本，其中完善性维护占比最大
2. 软件配置管理通过版本控制、变更控制保证软件资产的可追溯性
3. 项目估算方法多种多样，需结合项目特点选择合适的方法组合使用
4. 风险管理应贯穿项目全生命周期，做到"防患于未然"
