# 第五章 软件测试

## 学习目标
- 理解软件测试的目的与基本原则
- 掌握白盒测试与黑盒测试的常用方法
- 掌握测试的层次划分（单元、集成、系统、验收测试）
- 理解测试用例设计的常用技术
- 了解自动化测试与持续测试的实践

---

## 5.1 软件测试概述

### 5.1.1 测试的定义与目的
软件测试是为了发现程序中的错误而执行程序的过程。IEEE 定义：使用人工或自动化手段来运行或测定某个系统的过程，其目的在于检验它是否满足规定的需求或弄清预期结果与实际结果之间的差别。

**测试的核心目的**：**发现缺陷**，而不是证明程序没有错误（测试只能证明缺陷的存在，不能证明缺陷不存在）。

### 5.1.2 软件测试的基本原则
1. 所有测试都应追溯到用户需求
2. 应尽早并不断地进行测试
3. 完全测试是不可能的（穷举测试不可行）
4. Pareto 原则（80/20 原则）：80% 的缺陷往往集中在 20% 的模块中
5. 测试应从"小规模"开始，逐步扩展到"大规模"
6. 程序员应避免测试自己编写的程序（第三方测试更客观）

---

## 5.2 白盒测试（White-box Testing）

白盒测试又称结构测试，测试者了解程序的内部结构和逻辑，依据代码内部逻辑设计测试用例。

### 5.2.1 逻辑覆盖标准（从低到高）
1. **语句覆盖（Statement Coverage）**：每条语句至少执行一次
2. **判定覆盖/分支覆盖（Decision/Branch Coverage）**：每个判定的真假分支都至少执行一次
3. **条件覆盖（Condition Coverage）**：每个条件的可能取值至少出现一次
4. **判定/条件覆盖**：同时满足判定覆盖和条件覆盖
5. **条件组合覆盖（Multiple Condition Coverage）**：每个判定中条件的各种可能组合都至少出现一次
6. **路径覆盖（Path Coverage）**：覆盖程序中所有可能的执行路径（最严格）

### 5.2.2 基本路径测试法
基于程序的控制流图，计算**环形复杂度（Cyclomatic Complexity）**：
```
V(G) = E - N + 2
```
其中 E 为边数，N 为节点数。V(G) 决定了需要设计的独立路径测试用例数量下限。

## 5.3 黑盒测试（Black-box Testing）

黑盒测试又称功能测试，测试者不关心程序内部结构，只关注输入输出是否符合需求规格。

### 5.3.1 等价类划分法
将输入域划分为若干等价类，每类中选取一个代表性数据进行测试即可代表该类所有数据。
- **有效等价类**：符合需求规格、合理的输入
- **无效等价类**：不合理或非法的输入

**示例**：某字段要求输入 1-100 的整数
- 有效等价类：[1,100] 内的整数
- 无效等价类：小于1的数、大于100的数、非整数、空值

### 5.3.2 边界值分析法
经验表明，错误往往发生在输入域的边界上，而非中心。测试应重点关注边界值和边界值附近的数据（如上例中的 0、1、100、101）。

### 5.3.3 因果图法
适用于输入条件之间存在组合关系的场景，通过分析输入条件（因）与输出结果（果）之间的逻辑关系，生成判定表，设计测试用例，能够有效发现输入条件组合引起的问题。

### 5.3.4 场景法（用例场景法）
基于用例的基本流和备选流设计测试用例，覆盖正常业务流程和各种异常分支。

### 5.3.5 错误推测法
基于经验和直觉，推测程序中可能存在的错误类型，有针对性地设计测试用例（如输入空值、超长字符串、特殊字符）。

## 5.4 测试的层次

### 5.4.1 单元测试（Unit Testing）
针对最小可测试单元（函数、方法、类）进行测试，通常由开发人员编写，配合 xUnit 系列框架（JUnit、PyTest、Jest 等）。

### 5.4.2 集成测试（Integration Testing）
在单元测试基础上，将模块按照设计要求组装起来进行测试，主要发现模块间接口的问题。

**常见集成策略**：
- **大爆炸集成**：所有模块一次性集成（风险高，不推荐）
- **自顶向下集成**：从主控模块开始，逐步向下集成，需要使用**桩模块（Stub）**模拟未完成的下层模块
- **自底向上集成**：从底层模块开始集成，需要使用**驱动模块（Driver）**模拟上层调用者
- **三明治集成**：结合自顶向下和自底向上

### 5.4.3 系统测试（System Testing）
将整个软件系统作为一个整体，验证其是否满足系统规格说明，包括功能测试、性能测试、安全测试、兼容性测试、压力测试等。

### 5.4.4 验收测试（Acceptance Testing）
由用户或客户主导，验证系统是否满足业务需求，决定是否验收，常见形式：
- **Alpha 测试**：在开发环境下，由内部用户测试
- **Beta 测试**：在用户实际使用环境下测试

---

## 5.5 非功能性测试

- **性能测试（Performance Testing）**：验证系统在特定负载下的响应时间、吞吐量等指标
- **压力测试（Stress Testing）**：测试系统在超出正常负载、极限条件下的表现
- **安全测试（Security Testing）**：发现系统的安全漏洞（如 SQL 注入、XSS）
- **兼容性测试（Compatibility Testing）**：验证软件在不同浏览器、操作系统、设备上的表现
- **可用性测试（Usability Testing）**：评估软件的易用性和用户体验

---

## 5.6 自动化测试

### 5.6.1 自动化测试的优势
- 提高回归测试效率，节省人工重复劳动
- 支持持续集成/持续交付（CI/CD）流水线
- 提高测试的一致性和可重复性

### 5.6.2 测试金字塔（Test Pyramid）
```
        /\
       /UI\        少量：端到端测试（E2E），慢、脆弱、成本高
      /----\
     /集成测试\      适量：验证模块间协作
    /--------\
   /  单元测试  \    大量：快速、稳定、成本低，构成测试基础
  /------------\
```
**核心思想**：底层单元测试数量应最多，越往上层（集成、UI/E2E）数量应递减，以获得最佳的测试投入产出比。

### 5.6.3 常用自动化测试工具
- 单元测试：JUnit（Java）、PyTest（Python）、Jest（JavaScript）
- UI 自动化：Selenium、Playwright、Cypress
- 接口测试：Postman、RestAssured

---

## 本章小结
1. 白盒测试关注内部逻辑，黑盒测试关注外部行为，二者互为补充
2. 等价类划分和边界值分析是黑盒测试最常用、最有效的用例设计方法
3. 测试应分层进行：单元 → 集成 → 系统 → 验收，逐层扩大测试范围
4. 测试金字塔指导我们合理分配各层测试的比重，追求高效的质量保障体系
