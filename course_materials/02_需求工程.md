# 第二章 需求工程

## 学习目标
- 理解需求工程的过程与需求的分类
- 掌握需求获取的常用方法
- 掌握结构化分析方法（数据流图、数据字典）
- 掌握面向对象需求分析方法（用例建模）
- 理解需求规格说明书（SRS）的内容与作用

---

## 2.1 需求工程概述

### 2.1.1 什么是需求
需求是用户对目标软件系统在功能、行为、性能、设计约束等方面的期望。

### 2.1.2 需求的分类
1. **功能需求（Functional Requirements）**：系统必须完成的功能，例如"用户可以注册账号"
2. **非功能需求（Non-functional Requirements）**：
   - 性能需求（响应时间、吞吐量）
   - 可靠性需求（可用性、容错性）
   - 安全性需求
   - 可维护性、可扩展性需求
   - 易用性需求
3. **业务需求**：反映组织高层次的目标
4. **用户需求**：用户使用系统要完成的具体任务
5. **系统需求**：从系统角度描述软件、硬件应该实现的功能和特性

### 2.1.3 需求工程过程
需求工程通常包含以下活动：
```
需求获取 → 需求分析 → 需求规格说明 → 需求验证 → 需求管理
```

---

## 2.2 需求获取

### 2.2.1 常用需求获取方法
1. **访谈（Interview）**：与用户、领域专家一对一或小组交流
2. **问卷调查（Questionnaire）**：大范围收集用户意见
3. **观察（Observation）**：实地观察用户的工作流程
4. **原型法（Prototyping）**：构建软件原型让用户体验、反馈
5. **头脑风暴（Brainstorming）**：团队集思广益
6. **联合应用开发（JAD）**：组织用户、开发者共同参与的密集研讨会
7. **文档分析**：分析现有系统文档、业务流程规范

### 2.2.2 需求获取中的常见问题
- 用户不清楚自己真正需要什么（隐性需求）
- 用户需求经常变化
- 不同干系人（Stakeholder）需求存在冲突
- 沟通存在语言/专业术语障碍

## 2.3 结构化需求分析方法

### 2.3.1 数据流图（DFD, Data Flow Diagram）
数据流图用于描述系统中数据的流动、加工和存储，是结构化分析的核心工具。

**四种基本元素**：
| 符号 | 名称 | 含义 |
|------|------|------|
| 圆/圆角矩形 | 加工（Process） | 对数据进行的变换处理 |
| 箭头 | 数据流（Data Flow） | 数据的流动方向 |
| 双横线/开口矩形 | 数据存储（Data Store） | 数据保存的地方 |
| 矩形 | 外部实体（External Entity） | 系统之外的数据来源或去向 |

DFD 采用**分层细化**的方式：从顶层的 0 层图（系统上下文图）逐步分解为 1 层、2 层图，直至细化到基本加工。

### 2.3.2 数据字典（Data Dictionary）
对 DFD 中出现的所有数据流、数据存储、加工逻辑进行详细定义的工具，是结构化分析的另一个核心组成部分。

### 2.3.3 状态转换图（STD）
用于描述系统的状态及引起状态转换的事件，适合实时系统或反应式系统建模。

## 2.4 面向对象需求分析——用例建模

### 2.4.1 用例（Use Case）
用例描述了系统与外部参与者（Actor）之间的一次交互，代表参与者可以观察到的、有意义的系统功能单元。

### 2.4.2 用例图的组成要素
- **参与者（Actor）**：与系统交互的用户或外部系统
- **用例（Use Case）**：系统提供的功能
- **关联关系（Association）**：参与者与用例之间的连线
- **包含关系（Include）**：一个用例包含另一个用例的行为（必然发生）
- **扩展关系（Extend）**：一个用例在特定条件下扩展另一个用例的行为（可选发生）
- **泛化关系（Generalization）**：用例/参与者之间的继承关系

### 2.4.3 用例描述模板
一个完整的用例描述通常包含：
- 用例名称、编号
- 参与者
- 前置条件
- 主成功场景（基本流）
- 扩展场景（备选流/异常流）
- 后置条件

**示例：图书借阅用例**
```
用例名称：借阅图书
参与者：读者、图书管理员
前置条件：读者已登录系统，且无逾期未还图书
基本流：
  1. 读者搜索并选择要借阅的图书
  2. 系统检查图书库存是否充足
  3. 系统生成借阅记录
  4. 系统更新图书库存数量
  5. 系统提示借阅成功
异常流：
  2a. 若图书库存不足，系统提示"该书已全部借出"，流程结束
后置条件：借阅记录已生成，图书库存已更新
```

## 2.5 需求规格说明书（SRS）

### 2.5.1 SRS 的作用
- 作为开发方与客户方之间的**合同基础**
- 作为系统设计的**依据**
- 作为验收测试的**标准**

### 2.5.2 SRS 的主要内容（IEEE 830 标准参考结构）
1. 引言（目的、范围、术语定义）
2. 总体描述（产品概述、用户特点、约束条件）
3. 具体需求（功能需求、外部接口需求、性能需求、设计约束、质量属性）
4. 附录

### 2.5.3 需求的特性（好需求的标准）
一份高质量的需求应具备：
- **完整性（Complete）**
- **一致性（Consistent）**——需求之间不能互相矛盾
- **可行性（Feasible）**
- **必要性（Necessary）**
- **无二义性（Unambiguous）**
- **可验证性（Verifiable）**——需求可以通过测试来验证是否满足

---

## 2.6 需求验证与需求管理

### 2.6.1 需求验证
通过需求评审（Review）、原型验证等方式，检查需求文档是否正确反映了用户的真实意图。

### 2.6.2 需求变更管理
需求在项目生命周期中不可避免地会发生变化，需要建立正式的**变更控制流程**：
```
提出变更请求 → 影响分析（成本、进度、风险）→ 变更评审（CCB变更控制委员会）→ 批准/拒绝 → 实施变更 → 更新基线
```

---

## 本章小结
1. 需求工程是软件开发的起点，直接决定项目成败
2. 结构化方法（DFD+数据字典）适合早期面向过程的系统建模
3. 用例建模是面向对象需求分析的核心工具，强调"参与者-系统"的交互视角
4. 需求规格说明书是连接需求与设计的桥梁，需要满足完整性、一致性、可验证性等特性
