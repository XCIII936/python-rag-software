# 第三章 软件设计与架构

## 学习目标
- 理解软件设计的基本原则
- 掌握 SOLID 五大设计原则
- 理解常见软件架构风格（分层、MVC、微服务等）
- 了解常用设计模式的分类与典型应用场景
- 理解模块化设计中的耦合与内聚

---

## 3.1 软件设计概述

软件设计是将需求分析的结果转换为软件系统实现方案的过程，通常分为：
- **总体设计（概要设计）**：确定系统的总体架构、模块划分、模块间接口
- **详细设计**：确定每个模块的内部算法和数据结构

设计的核心目标：**高内聚、低耦合**，从而获得良好的可维护性、可扩展性和可复用性。

---

## 3.2 模块化设计：耦合与内聚

### 3.2.1 内聚（Cohesion）
内聚衡量模块内部各元素之间联系的紧密程度，内聚性从低到高分为：
1. 偶然内聚（最差）
2. 逻辑内聚
3. 时间内聚
4. 过程内聚
5. 通信内聚
6. 顺序内聚
7. 功能内聚（最佳）——模块内所有元素共同完成一个单一功能

### 3.2.2 耦合（Coupling）
耦合衡量模块之间相互依赖的程度，耦合性从低到高（越低越好）：
1. 无直接耦合（最佳）
2. 数据耦合
3. 标记耦合
4. 控制耦合
5. 外部耦合
6. 公共耦合
7. 内容耦合（最差）

**设计原则**：追求"高内聚、低耦合"，使模块易于理解、测试、修改和复用。

## 3.3 SOLID 设计原则

SOLID 是面向对象设计中五个基本原则的首字母缩写：

### S - 单一职责原则（Single Responsibility Principle）
一个类应该只有一个引起它变化的原因，即一个类只负责一项职责。

**反例**：一个 `User` 类既负责用户数据存储，又负责发送邮件通知——违反单一职责。
**改进**：拆分为 `User`（数据）和 `EmailNotifier`（发送邮件）两个类。

### O - 开闭原则（Open-Closed Principle）
软件实体（类、模块、函数）应该对扩展开放，对修改关闭。即通过新增代码而非修改已有代码来实现功能扩展，常通过抽象和多态实现。

### L - 里氏替换原则（Liskov Substitution Principle）
子类对象必须能够替换其父类对象，且不改变程序的正确性。即子类不应该违反父类的行为约定。

### I - 接口隔离原则（Interface Segregation Principle）
不应强迫客户依赖它们不使用的方法，应将庞大臃肿的接口拆分成更小、更具体的接口。

### D - 依赖倒置原则（Dependency Inversion Principle）
高层模块不应该依赖低层模块，二者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。这是实现"控制反转（IoC）"和依赖注入（DI）的理论基础。

## 3.4 软件架构风格

### 3.4.1 分层架构（Layered Architecture）
将系统划分为若干层，每层只能与相邻层通信，例如经典三层架构：
```
表示层（UI） → 业务逻辑层（BLL） → 数据访问层（DAL）
```
**优点**：职责清晰、易于维护
**缺点**：层间调用可能带来性能开销，牵一发而动全身的风险

### 3.4.2 MVC 架构模式（Model-View-Controller）
- **Model（模型）**：负责数据和业务逻辑
- **View（视图）**：负责界面展示
- **Controller（控制器）**：接收用户输入，调度 Model 和 View

**变体**：MVP（Model-View-Presenter）、MVVM（Model-View-ViewModel，常用于前端框架如 Vue/React 的响应式数据绑定）

### 3.4.3 客户端-服务器架构（C/S、B/S）
- **C/S（Client/Server）**：客户端需要安装专用程序
- **B/S（Browser/Server）**：客户端仅需浏览器，服务端集中部署，是当前 Web 应用的主流架构

### 3.4.4 微服务架构（Microservices）
将单一应用拆分为一组小型、独立部署的服务，每个服务：
- 围绕业务能力构建
- 可独立开发、部署、扩展
- 通过轻量级通信机制（如 HTTP/REST、消息队列）交互
- 通常搭配 API 网关、服务发现、容器化（Docker/K8s）技术

**优点**：技术栈灵活、故障隔离、独立扩展
**缺点**：分布式系统复杂度高（网络延迟、数据一致性、服务治理）

### 3.4.5 事件驱动架构（EDA）
系统组件通过发布/订阅事件进行通信，实现组件间的松耦合，适合异步、高并发场景。

### 3.4.6 管道-过滤器架构（Pipe-Filter）
数据在一系列处理单元（过滤器）之间通过管道传递，每个过滤器完成特定的数据变换，如编译器的词法分析→语法分析→代码生成流程。

## 3.5 常用设计模式简介

设计模式分为三大类（GoF 23 种设计模式）：

### 3.5.1 创建型模式（如何创建对象）
- **单例模式（Singleton）**：保证一个类只有一个实例，如数据库连接池
- **工厂方法模式（Factory Method）**：定义创建对象的接口，让子类决定实例化哪个类
- **抽象工厂模式（Abstract Factory）**：创建一系列相关或依赖对象的家族
- **建造者模式（Builder）**：将复杂对象的构建与表示分离

### 3.5.2 结构型模式（类和对象的组合）
- **适配器模式（Adapter）**：将一个类的接口转换为客户期望的另一个接口
- **装饰器模式（Decorator）**：动态地给对象添加额外的职责
- **代理模式（Proxy）**：为其他对象提供一种代理以控制对这个对象的访问
- **外观模式（Facade）**：为子系统中的一组接口提供一个统一的高层接口

### 3.5.3 行为型模式（对象间职责分配）
- **观察者模式（Observer）**：定义对象间一对多的依赖关系，当一个对象状态改变时，所有依赖者都会收到通知（发布-订阅的基础）
- **策略模式（Strategy）**：定义一系列算法，把它们封装起来，使它们可以互相替换
- **模板方法模式（Template Method）**：定义一个操作中的算法骨架，将某些步骤延迟到子类
- **责任链模式（Chain of Responsibility）**：将请求的处理者串成一条链，请求沿链传递直到被处理

## 本章小结
1. 良好的软件设计追求"高内聚、低耦合"
2. SOLID 原则是面向对象设计的基石，指导我们写出易维护、易扩展的代码
3. 架构风格的选择要结合系统规模、团队能力、非功能需求综合考量，微服务并非银弹
4. 设计模式是前人总结的、针对特定问题反复出现的通用解决方案
