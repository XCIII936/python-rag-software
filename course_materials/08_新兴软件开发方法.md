## 8.1 DevOps

### 8.1.1 什么是 DevOps
DevOps 是 Development（开发）与 Operations（运维）的组合词，是一种强调开发、测试、运维团队之间紧密协作、自动化、持续反馈的文化和实践集合，目标是缩短系统开发生命周期，实现持续交付高质量软件。

### 8.1.2 DevOps 的核心理念
1. **文化（Culture）**：打破开发与运维之间的壁垒，共担责任
2. **自动化（Automation）**：自动化构建、测试、部署流程
3. **度量（Measurement）**：通过数据驱动决策
4. **共享（Sharing）**：知识和工具的共享

### 8.1.3 DevOps 生命周期
```
计划 → 编码 → 构建 → 测试 → 发布 → 部署 → 运维 → 监控 → （反馈循环）
```

## 8.2 持续集成/持续交付/持续部署（CI/CD）

### 8.2.1 持续集成（CI, Continuous Integration）
开发人员频繁（每天多次）将代码集成到共享主干，每次集成都通过自动化构建（含测试）来验证，从而尽早发现集成错误。

**CI 关键实践**：
- 维护单一代码仓库
- 自动化构建
- 构建应包含自测试（单元测试）
- 每次提交都触发构建
- 保持构建速度快
- 任何人都可以看到最新构建结果

### 8.2.2 持续交付（CD, Continuous Delivery）
在持续集成的基础上，将集成后的代码自动部署到类生产环境，确保代码随时可以发布到生产环境，但生产环境的部署仍需人工审批触发。

### 8.2.3 持续部署（Continuous Deployment）
比持续交付更进一步，代码通过所有自动化测试后，自动部署到生产环境，无需人工干预。

### 8.2.4 CI/CD 流水线（Pipeline）示例
```
代码提交 → 触发流水线 → 代码编译 → 单元测试 → 代码质量扫描（SonarQube）
        → 构建镜像（Docker） → 集成测试 → 部署到预发环境 → 自动化验收测试
        → （人工审批）→ 部署到生产环境 → 监控告警
```

### 8.2.5 常用 CI/CD 工具
- Jenkins、GitLab CI/CD、GitHub Actions、CircleCI、Travis CI

## 8.3 云原生开发

### 8.3.1 什么是云原生（Cloud Native）
云原生是一套构建和运行应用程序的方法，充分利用云计算的弹性、分布式优势，核心技术包括：容器化、微服务、DevOps、持续交付。

### 8.3.2 容器化技术（Docker）
容器是一种轻量级的虚拟化技术，将应用及其依赖打包成一个可移植的镜像，实现"一次构建，到处运行"。

**核心概念**：
- **镜像（Image）**：只读的模板，定义容器运行环境
- **容器（Container）**：镜像的运行实例
- **Dockerfile**：描述如何构建镜像的脚本文件
- **Docker Compose**：用于定义和运行多容器应用的工具（本项目即采用此方式编排前后端与向量数据库服务）

### 8.3.3 容器编排（Kubernetes）
当容器数量增多、需要跨多台主机管理时，需要容器编排平台，Kubernetes（K8s）是目前事实上的标准：
- **Pod**：K8s 中最小的部署单元，包含一个或多个容器
- **Service**：为一组 Pod 提供稳定的网络访问入口
- **Deployment**：声明式管理 Pod 的副本数量和更新策略
- **自动扩缩容（HPA）**：根据负载自动调整 Pod 数量

### 8.3.4 微服务与云原生的关系
微服务架构是云原生应用的典型架构风格，配合容器化和 K8s 编排，可以实现服务的独立部署、弹性伸缩和故障自愈。

### 8.3.5 Serverless（无服务器架构）
开发者无需管理服务器，只需编写函数级别的代码（FaaS, Function as a Service），由云平台按需分配资源、按调用量计费，如 AWS Lambda、阿里云函数计算。

## 8.4 AI 辅助软件开发

### 8.4.1 AI 在软件开发生命周期中的应用
| 阶段 | AI 应用场景 |
|------|------------|
| 需求分析 | 从自然语言描述自动生成用户故事、需求文档摘要 |
| 设计 | 根据需求自动生成架构建议、API 设计 |
| 编码 | AI 代码补全与生成（GitHub Copilot、通义灵码等） |
| 测试 | 自动生成测试用例、智能缺陷预测 |
| 代码审查 | AI 辅助代码评审，发现潜在 Bug 和代码坏味道 |
| 运维 | AIOps：智能告警、异常检测、根因分析 |

### 8.4.2 大模型（LLM）驱动的智能体开发
以本课程项目为例，基于大语言模型（如通义千问 Qwen3-Max）构建的**智能体（Agent）**系统，通常包含以下核心能力：
1. **RAG（检索增强生成）**：结合向量数据库（如 Milvus）检索领域知识，增强大模型回答的准确性和专业性，减少幻觉
2. **工具调用（Function Calling）**：让大模型能够调用外部 API、数据库、代码执行环境完成复杂任务
3. **多轮对话与上下文管理**：维护会话历史，实现连贯的交互体验
4. **流式输出（SSE/WebSocket）**：提升大模型响应的实时性和用户体验

### 8.4.3 提示词工程（Prompt Engineering）
设计高质量的提示词是发挥大模型能力的关键技术，常见技巧：
- **角色设定**：明确告知模型扮演的角色（如"你是一位软件工程助教"）
- **少样本提示（Few-shot Prompting）**：给出示例引导模型输出格式
- **思维链（Chain-of-Thought）**：引导模型逐步推理，提高复杂任务的准确率
- **结构化输出约束**：要求模型以 JSON 等格式返回，便于程序解析

### 8.4.4 AI 辅助开发的挑战
- **代码正确性与安全性**：AI 生成的代码仍需人工审查，避免引入漏洞
- **知识产权与合规问题**
- **对开发者能力的依赖性**：AI 是辅助工具，不能替代扎实的工程基础

## 8.5 未来发展趋势展望

1. **低代码/无代码平台**：进一步降低软件开发门槛
2. **AI 原生应用（AI-Native App）**：以大模型为核心构建的新一代应用形态
3. **边缘计算与物联网软件工程**：软件工程实践向嵌入式、边缘设备延伸
4. **绿色软件工程**：关注软件系统的能耗与可持续性

## 本章小结
1. DevOps 通过文化变革和自动化工具链，打通了开发与运维的壁垒
2. CI/CD 流水线是现代软件工程实现"快速、可靠交付"的核心基础设施
3. 云原生技术（容器、微服务、K8s）为构建弹性、可扩展的现代应用提供了基础
4. 大模型和 AI 辅助开发正在深刻改变软件工程的实践方式，从编码辅助到智能体应用层出不穷
