## 4.1 编码规范

### 4.1.1 为什么需要编码规范
- 提高代码可读性，降低团队协作成本
- 减少低级错误，提高代码质量
- 便于代码维护和后续人员接手

### 4.1.2 常见编码规范内容
1. **命名规范**：
   - 变量、函数名做到见名知意（如 `getUserById` 而非 `func1`）
   - 遵循语言约定（如 Python 使用 `snake_case`，Java/JS 使用 `camelCase`，类名使用 `PascalCase`）
2. **格式规范**：缩进、空格、换行的一致性（通常借助 Prettier、Black、ESLint 等工具自动化）
3. **注释规范**：关键逻辑、复杂算法需要注释；函数需要文档字符串（docstring）说明参数、返回值
4. **异常处理规范**：不能吞掉异常、要给出有意义的错误信息
5. **禁止硬编码**：使用配置文件或常量管理魔法数字/字符串

## 4.2 代码重构（Refactoring）

### 4.2.1 什么是重构
在不改变软件外部行为的前提下，改善其内部结构，使代码更易理解、更易修改。

### 4.2.2 常见的代码坏味道（Code Smells）
| 坏味道 | 说明 |
|--------|------|
| 重复代码（Duplicated Code） | 相同或相似的代码出现在多处 |
| 过长函数（Long Method） | 函数职责过多，难以理解 |
| 过大的类（Large Class） | 类承担太多职责 |
| 过长参数列表（Long Parameter List） | 函数参数过多，调用复杂 |
| 霰弹式修改（Shotgun Surgery） | 修改一处功能需要在多个类中做出改动 |
| 依恋情结（Feature Envy） | 一个方法对另一个类的数据/方法过度依赖 |
| 数据泥团（Data Clumps） | 多个数据项总是一起出现，应抽取为对象 |

### 4.2.3 常见重构手法
- **提炼函数（Extract Method）**：将代码片段提炼成独立函数
- **内联函数（Inline Method）**：将过于简单的函数直接内联
- **提炼类（Extract Class）**：将类的部分职责拆分到新类
- **以多态取代条件表达式**：用继承/多态替代 if-else / switch 分支
- **引入参数对象（Introduce Parameter Object）**：将一组参数封装成对象

### 4.2.4 重构原则
- 重构前应有单元测试作为安全网
- 小步前进，每次只做一处修改，频繁验证
- 重构与新增功能应分开进行，不要混在一次提交中

## 4.3 版本控制（Git）

### 4.3.1 Git 基本概念
- **仓库（Repository）**：存放项目所有版本历史的数据库
- **工作区（Working Directory）**：本地文件系统中的项目文件
- **暂存区（Staging Area / Index）**：`git add` 后等待提交的改动
- **提交（Commit）**：一次代码快照，具有唯一哈希值
- **分支（Branch）**：独立的开发线，默认分支通常为 `main`/`master`

### 4.3.2 常用 Git 命令
```bash
git init                     # 初始化仓库
git clone <url>               # 克隆远程仓库
git add <file>                 # 添加到暂存区
git commit -m "message"       # 提交
git branch <name>             # 创建分支
git checkout -b <name>        # 创建并切换分支
git merge <branch>            # 合并分支
git rebase <branch>            # 变基
git push / git pull            # 推送/拉取远程仓库
```

### 4.3.3 常见分支模型
1. **Git Flow**：包含 `main`（生产）、`develop`（开发）、`feature/*`（特性）、`release/*`（发布）、`hotfix/*`（紧急修复）等分支
2. **GitHub Flow**：更轻量，`main` 为唯一长期分支，功能通过 `feature` 分支 + Pull Request 合入
3. **Trunk-Based Development（主干开发）**：所有开发者频繁地向主干（trunk）提交小的改动，配合特性开关（Feature Flag）

### 4.3.4 Pull Request / Merge Request 流程
```
创建功能分支 → 开发并提交 → 推送到远程 → 发起 PR/MR → 代码审查 → CI 自动化测试通过 → 合并入主分支
```

## 4.4 代码审查（Code Review）

### 4.4.1 代码审查的价值
- 提前发现 Bug 和设计缺陷
- 保证代码风格统一，符合团队规范
- 知识共享，帮助新人快速熟悉代码库
- 培养团队代码质量意识

### 4.4.2 代码审查的关注点
1. 功能正确性——是否实现了预期需求
2. 代码可读性——命名、结构是否清晰
3. 是否有安全隐患（如 SQL 注入、越权访问）
4. 是否有充分的单元测试覆盖
5. 是否遵循团队编码规范
6. 性能是否有明显问题（如 N+1 查询）

### 4.4.3 代码审查最佳实践
- 每次审查的代码量不宜过大（建议 < 400 行）
- 审查意见应具体、建设性，避免人身攻击式评论
- 结合自动化工具（Lint、静态扫描）减轻人工审查负担

## 4.5 单元测试

### 4.5.1 单元测试的意义
- 保证代码在修改后仍然正确（回归测试）
- 是重构的安全网
- 编写可测试的代码本身也会促使更好的设计（低耦合）

### 4.5.2 AAA 模式
一个良好的单元测试通常遵循 **Arrange-Act-Assert** 三段式结构：
```python
def test_calculate_discount():
    # Arrange：准备测试数据
    order = Order(amount=100)

    # Act：执行被测方法
    result = order.calculate_discount(rate=0.1)

    # Assert：验证结果
    assert result == 90
```

### 4.5.3 Mock / Stub
当被测代码依赖外部资源（数据库、网络请求、第三方 API）时，使用 Mock 对象模拟这些依赖，隔离被测单元，使测试快速、稳定、可重复执行。

### 4.5.4 测试覆盖率
测试覆盖率（Coverage）衡量测试用例覆盖代码的比例，常见指标：
- 语句覆盖率（Statement Coverage）
- 分支覆盖率（Branch Coverage）
- 路径覆盖率（Path Coverage）

**注意**：高覆盖率不等于高质量测试，覆盖率只是必要条件，不是充分条件。

## 本章小结
1. 编码规范和代码重构共同保障代码的可维护性
2. Git 是现代软件团队协作不可或缺的工具，理解分支模型有助于团队高效协作
3. 代码审查是质量保障的重要环节，兼具发现缺陷与知识传递的作用
4. 单元测试是持续交付高质量软件的基础设施
